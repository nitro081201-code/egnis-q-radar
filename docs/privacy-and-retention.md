# 개인정보 및 데이터 보존 정책 (Gate 1)

## 1. 저장하지 않는 개인정보

다음 항목은 어떤 테이블에도, 어떤 형태로도 저장하지 않습니다.

- 대표자명
- 전화번호 / 휴대전화번호
- 이메일(사용자 계정 이메일 제외)
- 상세주소 (시/군/구 이하)
- 전체 인허가번호 원문

## 2. 대체 저장 방식

| 원본 항목 | 저장 방식 | 컬럼 |
|---|---|---|
| 상세주소 | 시/도 단위까지만 | `region` |
| 인허가번호 | 화면 표시용 마스킹(예: `2019-****`) + 중복확인용 단방향 해시. 원본은 해시 생성 후 즉시 폐기 | `license_no_masked`, `license_no_hash` |
| API 원문 응답 | 개인정보 제거 + 허용 필드만 남긴 값만 저장 (필드 전체 무손실 저장 금지) | `sanitized_raw_data` |

`sanitized_raw_data`는 공공데이터 API가 응답한 원문 그대로가 아니라, 애플리케이션 코드가
허용 목록(allowlist)에 있는 필드만 골라 담은 값입니다. 이 저장 로직과 개인정보 제거 로직은
Gate 3(공식 API 연결)에서 구현하며, 자동 테스트(§21 테스트 항목 1, 2)로 검증합니다.

## 3. 행정처분 공개기한 관리 (`dispositions` 테이블)

| 컬럼 | 의미 |
|---|---|
| `public_until` | 공식 공개기한. 값이 없거나 파싱 실패 시 자동 공개하지 않고 검수대상으로 둡니다. |
| `source_updated_at` | 공식 데이터 최종수정일 |
| `first_seen_at` / `last_seen_at` | 최초 수집시각 / 마지막 확인시각 |
| `visibility_status` | `visible`(공개) / `expired`(공개기한 만료) / `quarantined`(검수대상) / `rejected`(노출 부적절) |

공개기한이 지나면 일반 사용자 화면(목록/검색/보드/CSV)에서 업체 식별정보가 자동으로
사라집니다. 관리자만 만료 여부와 내부 조치 존재 여부를 계속 확인할 수 있습니다.

**공개기한 만료 데이터를 삭제할지 계속 보관할지는 아직 정책이 정해지지 않았습니다.**
임의로 영구 보존 정책을 만들지 않았으며, 필요 시 Jay님의 결정을 받아 반영합니다.

## 4. 데이터 상태 (`status` 컬럼 — 모든 콘텐츠 테이블 공통)

`ingested`(수집됨) → `quarantined`(검수대상) / `unverified`(수동등록 검수대기) →
`published`(공개) → `expired`(공개기한 만료) / `rejected`(오등록·중복 등으로 노출 부적절)

**"API로 수집됐다고 무조건 화면에 노출되지 않습니다."** 업체명·제품명 누락, 고유키 생성 실패,
날짜 파싱 실패, 개인정보 제거 실패, 공개기한 확인 실패, 원문값 표준화 실패 중 하나라도
있으면 `quarantined` 상태로 남고 일반 화면에는 보이지 않습니다. 이 건수는 관리자 화면에서
확인할 수 있습니다 (Gate 2에서 구현).

## 5. 나중에 Supabase 대시보드에서 직접 확인하는 방법 (Gate 1 완료 후)

1. Supabase 대시보드 → Table Editor
2. `dispositions` 테이블을 열어 컬럼 목록에 대표자명·전화번호·상세주소·전체 인허가번호
   컬럼이 **없는지** 확인
3. `public_until`, `visibility_status` 컬럼이 **있는지** 확인
4. `profiles` 테이블에 `role`, `is_active` 컬럼이 있는지 확인
